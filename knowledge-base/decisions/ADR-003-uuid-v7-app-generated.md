# ADR-003 ‚Äî UUID v7 as primary key, generated in application code

- **Status:** Accepted
- **Date:** 2026-05-26
- **Deciders:** Tech lead (MotivesVN IT)
- **Related:** `02-tech-stack.md` (Database section), `04-data-model.md`, `08-coding-standards.md`, `backend/src/Rmms.Domain/Common/UuidV7.cs`, `infra/postgres/init/01-extensions.sql`

## Context

Every aggregate root in RMMS needs a primary key. The choice affects:

- **Insert hot-spot behavior** (random GUIDs cause B-tree page splits and degrade insert throughput on tables with millions of rows).
- **Client-side ID generation** (mobile app needs to mint IDs offline for form drafts per `06-business-rules.md`, where Phase 1 allows offline form drafts only).
- **Idempotency** (the `X-Idempotency-Key` header pattern per `05-api-conventions.md` benefits when the client can attach a stable ID before the round trip).
- **External integration** ‚Äî IDs may appear in URLs, logs, support tickets; non-secrets but should be **non-guessable** and **not enumerable** (so plain `bigint` sequences are out).

Options on the table at scaffold review:

- `bigserial` (PostgreSQL auto-increment) ‚Äî fast inserts, enumerable IDs, no client-side generation.
- `uuid` (v4 random) ‚Äî non-enumerable, but B-tree-unfriendly; PostgreSQL has `gen_random_uuid()` in `pgcrypto`.
- `uuid` (v7 time-ordered) ‚Äî non-enumerable AND B-tree-friendly. **Spec is RFC 9562 (published May 2024).**

PostgreSQL 16 (our target ‚Äî see `02-tech-stack.md`) **does not ship `uuidv7()` natively**. Options to obtain v7 IDs:

1. Install the `pg_uuidv7` extension on Postgres.
2. Generate IDs in the application using a C# implementation of RFC 9562.

## Decision

**Use UUID v7 as the primary key type for all aggregate roots. Generate them in C# code, not in the database.**

Concretely:

- All aggregate roots (and most child entities where independent identity matters) use `Guid` properties named `Id` mapped to `uuid` Postgres columns.
- The implementation lives at `backend/src/Rmms.Domain/Common/UuidV7.cs` and exposes a static `UuidV7.New()` that returns a `Guid` whose first 48 bits encode Unix-ms-time and the rest is RFC-9562-compliant randomness.
- `Entity<TId>` (and `AggregateRoot<TId>`) initialize `Id = UuidV7.New()` in their default constructor.
- No PostgreSQL extension is required for ID generation. `pgcrypto` and `citext` are enabled for other reasons (see `infra/postgres/init/01-extensions.sql`); `pg_uuidv7` is **not** installed.
- Existing reference data (e.g., role enum lookup tables) may continue to use small string codes (`"pg"`, `"leader"`) rather than UUIDs ‚Äî UUIDs are only for entities with mutable lifecycle.

## Alternatives considered

1. **`bigserial` / `bigint identity`**
   - Pros: fastest inserts; tiny indexes; debuggable; PostgreSQL native.
   - Cons: enumerable (user 1, 2, 3 reveals system size); cannot mint client-side for offline form drafts; harder to merge data across environments (collisions). Stronger argument against: external systems (FPT.AI, SendGrid) may surface our IDs in webhooks/logs ‚Äî enumerable internal IDs leak business metrics.
   - **Rejected.**

2. **UUID v4 (random) with `gen_random_uuid()`**
   - Pros: trivially supported by `pgcrypto`; non-enumerable; widely understood.
   - Cons: random distribution causes B-tree page splits on insert ‚Üí 30‚Äì50% slower bulk inserts on `attendance_records`, `audit_log` (write-heavy tables); higher WAL volume; harder to debug "what was inserted around 10am" by scanning IDs.
   - **Rejected** because the write-heavy tables (`attendance_records`, `audit_log`, `form_submissions`) will dominate cost.

3. **UUID v7 generated by `pg_uuidv7` extension**
   - Pros: PostgreSQL handles all generation; consistent across any client; less code to maintain.
   - Cons: extension is third-party (not in `contrib`); requires `CREATE EXTENSION` privilege; complicates managed-Postgres migration (some cloud Postgres services don't allow custom extensions); cannot generate IDs client-side (offline form drafts can't have stable IDs until they reach the server).
   - **Rejected.**

4. **UUID v7 generated in C# application code**
   - Pros: no Postgres extension dependency ‚Üí portable across any Postgres provider; client (web/mobile) can mint IDs before the round-trip ‚Üí enables true offline form drafts (Phase 1 scope) and natural idempotency; B-tree-friendly insert order; RFC 9562 standard. .NET 9+ ships `Guid.CreateVersion7()` but our `UuidV7` helper predates and abstracts it for cross-version stability.
   - Cons: ~50 LOC of C# code we own; mobile (Flutter) and web (TypeScript) need their own RFC 9562 implementations to mint IDs locally (lightweight ‚Äî a few small packages exist).
   - **Accepted.**

## Consequences

**Positive**

- Insert performance on write-heavy tables (`attendance_records`, `audit_log`, `form_submissions`) stays close to `bigserial` because IDs increase nearly monotonically.
- Offline form drafts on mobile (per Phase 1 partial-offline scope) get stable IDs at draft-create time ‚Üí server-side dedupe is trivial.
- IDs in URLs/logs are non-enumerable (no `?id=1234` enumeration attacks).
- No PostgreSQL extension dependency ‚Üí managed-Postgres migration (RDS, Cloud SQL, etc.) is open.

**Negative / accepted trade-offs**

- IDs are 128 bits ‚Üí indexes and FKs are 2√ó the size of `bigint`. At RMMS scale (one customer, ‚â§500 users, ‚â§100k attendance records/month) this is negligible.
- Slightly slower exact-id lookups vs `bigint` due to larger key size. Negligible at our scale; offset by being B-tree-ordered.
- Reading a UUID v7 is harder than `12345` for humans ‚Äî mitigated by truncated display (`d6f8‚Ä¶2c91`) in admin views.

**Mitigations**

- Provide a `UuidV7.Parse(string)` and `ToString()` helper that always normalizes lowercase hex with hyphens ‚Äî consistent serialization across DB, JSON, logs.
- For high-cardinality columns where lookup speed matters (e.g., `audit_log.user_id`), add a composite index `(user_id, created_at DESC)` rather than relying on the PK index alone.

## Revisit triggers

- Microsoft .NET BCL deprecates `Guid` in favor of a new struct (extremely unlikely in 10+ years).
- A managed-Postgres provider we target adds first-class `uuidv7()` and removes the cost of relying on it.
- Insert performance on `audit_log` or `attendance_records` becomes the bottleneck despite UUID v7 (then revisit table partitioning or move to `bigserial` for purely-internal logs).
